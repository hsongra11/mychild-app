## 2. Methodology

### 2.1 Autoresearch Approach

This research adapts Karpathy's [autoresearch](https://github.com/karpathy/autoresearch) pattern — originally designed for neural network architecture search — to rule-based developmental screening engine optimization.

**Key adaptation:** Because the mychild-engine runs in milliseconds (not minutes like ML training), experiments complete 100-1000x faster. This enabled 325 threshold experiments and 70,192 adversarial profile generations in under 4 hours of compute time across 4 runs.

**Architecture (Run 4):** Single-session Claude core loop with:
- `experiment-runner.js` — threshold mutation with auto-revert in `finally` block
- `adversarial-phase2.ts` — bounded adversarial profile generation
- `autoresearch-checkpoint.ts` — compact summary generation
- Checkpoint worker running in parallel for periodic metric refresh

### 2.2 Validation Framework

Two validation sets with distinct purposes:

**Curated set (N=12,991 observations, 1,747 profiles):**
- 108 original hand-labeled profiles with expert-verified ground truth
- 1,639 promoted adversarial profiles where engine output matched clinical ground truth with high confidence
- Purpose: regression testing, internal consistency verification
- Ground truth: hand-verified (original) + engine-agreeing with clinical expectations (promoted)

**Adversarial set (N=21,596 observations, 70,192 raw profiles):**
- Generated by autonomous AI across 18 categories targeting engine boundaries
- Ground truth methodology: **run-then-label** — profiles are run through the engine first, then an AI evaluator judges whether the engine's output is clinically reasonable
- Purpose: stress testing, boundary discovery, honest performance measurement
- 315 critical findings (engine missed a real concern), 9 notable findings (false positives), 89 minor findings

### 2.3 Ground Truth Methodology

**Run-then-label (adversarial set):** Unlike traditional predict-then-compare, the adversarial evaluator:
1. Generates a profile without preconceptions
2. Runs it through the engine
3. Reads the engine's classification
4. Judges: "Is this clinically reasonable?" based on CDC 2022 milestones, normative age expectations, and developmental science

This eliminates prediction bias — the evaluator doesn't have to guess what the engine will do.

**Limitations of AI-generated ground truth:**
- The evaluator is an AI, not a clinician. Its clinical judgment is approximate.
- Systematic biases may exist (e.g., AI may be more conservative than clinicians)
- Ambiguous cases (borderline presentations, sparse data) have uncertain labels
- Ground truth quality varies by category — "clear typical" and "clear delay" labels are highly reliable; "borderline" and "sparse" labels are less certain

### 2.4 Statistical Methods

- **Cohen's kappa (κ):** Chance-corrected agreement measure. Interpretation: <0.20 poor, 0.21-0.40 fair, 0.41-0.60 moderate, 0.61-0.80 substantial, 0.81-1.00 almost perfect.
- **Sensitivity:** True positive rate (TP / (TP + FN)). Proportion of true concerns correctly identified.
- **Specificity:** True negative rate (TN / (TN + FP)). Proportion of typically-developing children correctly cleared.
- **PPV/NPV:** Positive and negative predictive values. Probability that a flagged/cleared result is correct.
- All observations are at the domain level (8 domains per profile), not profile level.

### 3.3 Disagreement Analysis

Of 781 adversarial disagreements:

**False negatives (engine missed concern):** The majority of disagreements. The engine tends to under-flag, particularly in:
- Sparse data scenarios where few questions are answered
- Borderline cases near grace window boundaries
- Single-domain delays where only 1-2 milestones are missed

**False positives (engine over-flagged):** Fewer cases. Engine sometimes flags concern when:
- Child is preterm and milestones are within corrected age expectations
- Only low-weight milestones are missed within the grace window

**Clinical significance:** The engine's bias toward under-flagging (false negatives > false positives) means some real concerns may be missed. For a screening tool, the ideal bias is the opposite — better to over-flag (refer for evaluation) than miss. This is the primary area for improvement.

## 7. Engine Improvements (Runs 1-3)

14 engine improvements were committed across the first 3 runs. All maintain 100% curated sensitivity and are grounded in CDC 2022 milestones.

### Run 1 (5 changes)
1. **RF domain flag downgrade prevention** — prevents RF-sourced flags from being downgraded by grace windows
2. **Single-flag domain downgrade with strong normal evidence** — single flag in a domain with many achieved milestones downgrades to watch status
3. **Cross-domain correction for RF-sourced flags** — handles RF flags that appear in non-RF domains

### Run 2 (5 changes)
1. **Moderate concern threshold adjustment** — totalWeightedPoints >= 3 triggers moderate_concern
2. **Lower totalWeightedPoints threshold** — threshold lowered to 2 for moderate_concern

### Run 3 (4 changes)
1. **RF domain scoring** — RF scored as independent domain, reducing false negatives from 101 to 48
2. **Evidence gate tuning** — adjusted sufficiency gates for better sparse-data handling
3. **Low_concern binary inclusion** — low_concern classified as "concern" in binary threshold (conservative screening)
4. **Sparse precaution downgrade** — prevents false flags when evidence is insufficient

## 8. Limitations

### 8.1 Synthetic Validation Only

All metrics are from synthetic profiles, not clinical data. Synthetic validation demonstrates internal consistency and algorithmic correctness but cannot substitute for clinical validation against:
- Clinician-adjudicated developmental assessments
- Standardized diagnostic instruments (Bayley, WISC, etc.)
- Real-world population distributions and prevalence rates

### 8.2 AI-Generated Ground Truth

Adversarial profile ground truth is generated by an AI evaluator, not a clinician. Known limitations:
- AI may systematically over- or under-estimate clinical significance
- Ambiguous cases (borderline, sparse) have uncertain labels
- Cultural and demographic factors are not modeled
- The evaluator uses CDC 2022 guidelines but may misapply them in edge cases

### 8.3 Curated Set Construction Bias

The expanded curated set (1,747 profiles) includes 1,639 promoted adversarial profiles where the engine already agreed with ground truth. This creates selection bias — the curated set over-represents cases the engine handles well. The kappa = 1.000 is a consequence of this construction, not evidence of perfection.

### 8.4 Population Representation

Synthetic profiles do not represent real-world population distributions:
- Prevalence of developmental delays in the adversarial set (~15%) is higher than population prevalence (~5-15%)
- Preterm and regression profiles are over-represented relative to clinical populations
- Cultural variation in milestone achievement timing is not modeled

### 8.5 Single-Engine Validation

All experiments test a single configuration of the rules engine. No comparison against alternative rule sets, ML-based approaches, or ensemble methods was performed.

## 9. Recommendations

### 9.1 Priority 1: Clinical Validation

The next essential step is validation against real patient data with clinician-adjudicated outcomes. Recommended approach:
- Partner with 2-3 pediatric clinics
- Collect parent-reported milestone data via the engine
- Compare engine classifications against clinician developmental assessments
- Target N >= 500 children across the 0-36 month range
- Pre-register the study protocol

### 9.2 Priority 2: Address False Negatives

The engine's primary weakness is under-flagging (86.5% sensitivity). Specific areas:
- Improve sparse-data handling — when few questions are answered, consider flagging for "insufficient data" rather than "no concern"
- Review grace window logic for single-domain delays
- Consider domain-specific scoring rules for high-risk domains (EL, SE)

### 9.3 Priority 3: Triage Critical Adversarial Findings

315 critical findings from the adversarial set need human review to determine:
- Which represent genuine engine gaps (fix needed)
- Which represent AI evaluator errors (ground truth is wrong)
- Which represent ambiguous cases (no clear right answer)

### 9.4 Priority 4: Expand Question Bank

Current validation is limited by the 131-question bank. More questions per domain would enable:
- Finer-grained age band coverage
- Better sparse-data resilience
- More discriminating scoring

### C. Glossary

- **Curated set:** Hand-verified profiles used for regression testing. kappa = 1.000 expected by construction.
- **Adversarial set:** AI-generated stress-test profiles with run-then-label ground truth. The primary measure of engine quality.
- **Run-then-label:** Ground truth methodology where the engine runs first, then a clinical evaluator judges correctness. Eliminates prediction bias.
- **Grace window:** Time buffer (in weeks) before a missed milestone is counted as concerning. Accounts for normal developmental variation.
- **Weight class:** Question importance: L (low), M (medium), H (high), RF (red flag). RF milestones always trigger concern.
- **Corrected age:** For preterm infants, milestones are evaluated against adjusted age (chronological age minus weeks premature) up to 24 months.
